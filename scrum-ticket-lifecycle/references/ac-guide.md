# AC 定義規範

## 格式選擇

有使用者行為流程 → 用 **Given / When / Then**
```
Given [初始狀態或前提]
When  [使用者執行某個動作]
Then  [系統應該有什麼反應]
```

條件明確、無需描述流程 → 用 **Checklist**
```
- [ ] 條件一（可驗證的事實）
- [ ] 條件二
```

---

## 好 AC 的三個判斷標準

| 標準 | 說明 | 反例 → 正例 |
|------|------|------------|
| **可驗證** | 能用 Yes/No 判斷是否達成 | ❌「系統要跑得快」→ ✅「頁面載入 < 3 秒」 |
| **有邊界** | 明確範圍，不留灰色地帶 | ❌「顯示旅客資訊」→ ✅「顯示姓名、護照號碼、國籍」 |
| **不含實作** | 說「達成什麼」，不說「怎麼做」 | ❌「用 Redis 快取配額」→ ✅「配額查詢回應 < 500ms」 |

---

## AC 必須覆蓋

- Happy path（正常流程）
- 至少一個 edge case 或異常處理（如：配額不足、權限不符、資料為空）

**AC 數量：每張 Story 建議 2–5 條。**
超過 5 條通常代表 Story 太大，應考慮拆卡。

---

## 常見錯誤（寫了等於沒寫）

- ❌ 把 AC 寫成 Task：「工程師需要串接 S6 的訂單 API」
- ❌ AC 過於模糊：「使用者可以順利完成報名流程」
- ❌ 只有 happy path，沒有異常處理
- ❌ AC 內容與 Story 描述完全重複，沒有增加資訊
